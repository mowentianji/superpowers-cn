## 审查清单

**代码质量：**
- 职责分离是否清晰？
- 错误处理是否正确？
- 是否具备类型安全（如适用）？
- 是否遵循 DRY 原则？
- 是否处理了边界情况？

**架构：**
- 设计决策是否合理？
- 是否考虑了扩展性？
- 是否存在明显的性能影响？
- 是否引入安全问题？

**测试：**
- 测试是否真的验证了业务逻辑，而不是只验证 mock？
- 边界场景是否覆盖？
- 是否需要补充集成测试？
- 所有测试是否通过？

**需求符合性：**
- 是否满足计划中的全部要求？
- 实现是否符合规格？
- 是否出现范围蔓延？
- 重大变更是否已记录？

**生产可用性：**
- 如果架构发生变化，是否考虑了迁移策略？
- 是否考虑了向后兼容？
- 文档是否完整？
- 是否还存在明显 Bug？

## 关键规则

**要做：**
- 按真实严重性分类，不要夸大
- 具体到文件和行号，不要模糊表述
- 解释问题为什么重要
- 承认优点，不要只挑错
- 给出明确结论

**不要：**
- 在没认真检查的情况下说“看起来不错”
- 把吹毛求疵的问题标成严重
- 对没有审阅的代码发表结论
- 给出模糊建议（例如“改进错误处理”）
- 回避明确判断
